# FOMO — AI热点资讯自动采集 Chrome 浏览器扩展

## 一、项目目标与总体架构

FOMO 是一款 Chrome 浏览器扩展，其核心功能是通过 AI 智能体（Agent）自动在互联网上搜索并采集当前最热门的人工智能相关资讯，包括开源项目、学术论文、业界讨论等。用户只需在浏览器中点击一个按钮，扩展便会接管当前标签页，自主导航到 Twitter/X、GitHub Trending、ArXiv、Reddit 等平台，像一个真人一样浏览网页、点击链接、滚动页面、提取信息，最终收集约二十条高质量的 AI 资讯并以结构化数据供用户下载。

整体架构遵循 Chrome Manifest V3 扩展规范，分为三大层：

- **弹出窗口层（Popup）**：极简的用户交互界面，展示任务进度和控制按钮。
- **后台服务层（Background Service Worker）**：扩展的核心运行时，承载 AI 智能体的执行循环、浏览器自动化控制、大模型调用。
- **类型定义层（Types）**：全局共享的数据结构定义，保证前后端数据一致性。

这三层通过 Chrome 扩展消息机制通信：Popup 发送指令（开始、停止、查询状态、下载结果），Background 执行任务并将进度实时推送回 Popup。

## 二、技术栈与构建配置

### 语言与运行环境

项目采用 TypeScript 编写，编译目标为 ES2022，使用 ESNext 模块系统。TypeScript 开启了严格模式，同时引入了 DOM、DOM.Iterable 以及 Chrome 扩展类型定义。项目声明了三种类型引用：Chrome 扩展 API 类型、Vite 客户端环境类型、以及 Node.js 类型。

### 构建工具

使用 Vite 5 作为构建工具。Vite 配置了多入口打包策略：一个入口是 Popup 的 HTML 页面，另一个入口是 Background 的 TypeScript 主文件。构建时禁用了代码压缩以便于调试，并生成了 Source Map。构建目标设置为最新的 ESNext 标准。

Vite 配置中有一个自定义插件，在构建完成后执行以下收尾操作：将扩展清单文件复制到输出目录；将 Popup 的样式文件复制到输出目录的对应位置；将图标目录下的所有 PNG 和 SVG 图片文件复制到输出目录的图标子文件夹中。

构建输出的文件组织方式经过精心设计：Background 入口打包后输出到输出目录下的 background 子目录，命名为 index.js；Popup 相关文件按原有目录结构输出；CSS 文件统一输出到 popup 子目录下；其他代码块输出到 chunks 子目录并附带哈希值。

Rollup 配置将 chrome 全局对象和所有 Node 内置模块标记为外部依赖，不打包进最终产物。

### 环境变量注入

项目使用 Vite 的环境变量机制，通过 loadEnv 函数加载环境配置。三个关键环境变量在构建时被静态替换到代码中：大模型 API 密钥、API 基地址、以及模型名称。这些变量从项目根目录的环境配置文件（如 .env.local）中读取。

### 依赖关系

运行时依赖仅有 openai 这一个库，版本为 4.28 及以上。它被用作通用的 OpenAI 兼容 API 客户端，实际连接的是阿里云百炼平台（DashScope）。开发依赖包括 Chrome 扩展类型定义、Node.js 类型定义、TypeScript 编译器以及 Vite 构建工具。

### 构建脚本

项目提供两个构建命令：dev 命令启动 Vite 的监听模式构建，用于开发时自动重编译；build 命令执行一次性生产构建。

## 三、目录结构与文件命名约定

项目根目录名为 fomo，其下有以下结构：

- **根目录配置文件**：包括包管理配置文件、TypeScript 编译配置文件、Vite 构建配置文件、Chrome 扩展清单文件。
- **types.ts**：位于根目录，是全局共享的类型定义文件。
- **background 目录**：后台服务相关代码，是项目的核心业务逻辑所在。入口文件为该目录下的 index.ts。其下再分为三个子目录：
  - **agent 子目录**：AI 智能体相关逻辑，包含执行器（executor.ts）、提示词（prompts.ts）和工具定义（tools.ts）三个文件。
  - **browser 子目录**：浏览器自动化控制，包含浏览器上下文管理（context.ts）和页面控制器（page.ts）两个文件。
  - **llm 子目录**：大模型客户端，仅含一个文件（dashscope.ts）。
- **popup 目录**：弹出窗口界面，包含 HTML 页面文件、CSS 样式文件和 TypeScript 交互逻辑文件。
- **icons 目录**：存放扩展图标，支持三种尺寸（16像素、48像素、128像素）的 PNG 图片。

命名约定上，所有文件均使用小写字母加连字符（kebab-case）或全小写单词的形式。TypeScript 文件使用 .ts 后缀。每个目录的入口文件统一命名为 index.ts。子模块按职责命名，如 executor、prompts、tools、context、page、dashscope。

## 四、Chrome 扩展清单配置

扩展使用 Manifest V3 规范。名称为"FOMO - AI热点搜索"，版本号 1.0.0。

扩展声明了以下权限：tabs（标签页管理）、scripting（脚本注入执行）、storage（本地存储）、downloads（文件下载）、activeTab（当前活动标签页访问）。同时声明了全 URL 的主机权限，使扩展能在任意网站上注入脚本。

后台服务声明为模块类型的 Service Worker，入口指向构建后的 background/index.js。扩展的浏览器操作按钮关联一个弹出页面和一个默认提示文字。

## 五、各模块职责与接口详述

### 5.1 类型定义模块（types.ts）

此文件定义了整个项目中所有模块共享的数据结构，不包含任何业务逻辑。

**搜索结果结构**描述一条被采集到的 AI 资讯，包含以下字段：资讯标题（必填）、信息来源 URL（必填）、产品主页链接（可选）、GitHub 仓库链接（可选）、ArXiv 论文链接（可选）、创新点描述（必填，中文）、开发公司或团队名称、热度趋势等级（高、中、低三档之一）、标签数组、来源平台名称、以及发现时间的 ISO 时间戳。

**智能体状态结构**维护 Agent 运行时的全部状态信息：已收集的结果数组、已访问过的 URL 集合（用于去重）、当前执行步骤编号、最大步骤数上限、目标采集数量、以及一个布尔标记表示任务是否完成。

**执行状态枚举**有五个取值：空闲、运行中、暂停、已完成、出错。

**状态更新结构**是从 Background 推送到 Popup 的实时状态快照，包含执行状态、当前进度数、总目标数、当前正在执行的动作描述（可选）、以及错误信息（可选）。

**日志条目结构**记录 Agent 执行过程中的每一步详情，包含时间戳、日志类型（助手消息、工具调用、工具响应、用户消息、系统事件、错误共六种）、步骤编号、以及任意格式的内容。

**执行日志结构**是完整的一次任务执行记录，包含开始时间、结束时间（可选）、总步骤数、结果数量、以及日志条目数组。

**消息类型联合类型**定义了 Popup 与 Background 之间所有可能的通信消息格式，共八种：启动任务（附带标签页 ID）、停止任务、查询状态、下载结果、下载日志、状态更新推送、任务完成推送、错误推送。

### 5.2 后台服务入口模块（background/index.ts）

这是 Chrome 扩展的 Service Worker 入口，负责：

**状态管理**：维护四个模块级变量——当前执行器实例、当前已收集的结果数组、当前执行状态、以及当前执行日志。

**消息通信**：提供三个向 Popup 发送消息的函数——发送状态更新、发送任务完成通知（附带结果数组）、发送错误通知。所有发送操作都静默捕获错误，因为 Popup 窗口可能未打开。

**任务启动流程**：接收标签页 ID 后，先检查是否已有任务在运行（如有则忽略），然后初始化浏览器上下文并设置当前标签页，尝试连接该标签页（若无法连接则抛出异常），创建 Agent 执行器实例（目标数量 20 条、最大步骤 100 步），注册状态回调函数，运行执行器主循环，获取执行日志，更新状态为已完成并通知 Popup。若过程中出错，状态标记为错误并向 Popup 发送错误信息。无论成功与否，最终都会清理浏览器上下文并释放执行器。

**任务停止**：调用执行器的取消方法，清空执行器引用，重置状态为空闲，清理浏览器上下文。

**消息监听**：注册了一个统一的消息监听器，根据消息类型路由到对应处理函数。支持的消息类型包括：启动任务、停止任务、获取状态（同步返回当前进度快照）、下载结果（同步返回结果数组）、下载日志（同步返回执行日志）。监听器始终返回 true 以保持消息通道开启支持异步响应。

**事件监听**：监听标签页关闭事件，当标签页被关闭时从浏览器上下文中移除对应页面。监听扩展安装或更新事件，用于日志记录。

### 5.3 AI 智能体执行器模块（background/agent/executor.ts）

这是整个项目最核心的模块，实现了 ReAct（Reasoning + Acting）循环，即大模型在"思考-行动-观察"之间不断迭代的自主执行模式。

**初始化**：构造函数接收两个参数——目标采集数量（默认 20）和最大步骤数（默认 100）。初始化时创建 Agent 状态对象，构建初始消息列表（包含系统提示词和初始用户指令），创建工具执行器实例，并初始化执行日志。

**主循环逻辑**（run 方法）：

循环条件为：任务未完成、未被取消、且当前步骤未超过最大步骤限制。每次循环执行以下操作：

第一步，递增步骤计数器。

第二步，将当前完整消息历史发送给大模型，同时传入可用工具列表，让大模型决定下一步行动。

第三步，检查大模型响应。如果响应为空，记录错误日志并跳过当前步骤继续下一轮。

第四步，如果大模型返回了文本内容（即"思考"部分），将其截取前一百个字符作为当前动作状态发送给 Popup 展示。

第五步，将大模型的完整响应（包含文本和工具调用指令）添加到消息历史中，并写入执行日志。

第六步，如果大模型请求了工具调用，依次执行每个工具调用：解析工具名和参数（JSON 解析失败时使用空对象），调用工具执行器执行对应操作，将工具返回结果添加到消息历史中（使用 tool 角色和对应的调用 ID）。工具响应在写入日志时，如果超过五千字符会被截断。

第七步，判断任务是否完成。如果大模型选择了自然停止（finish_reason 为 stop 且没有工具调用），但已采集结果数量不足目标数，则向消息历史追加一条用户消息，告知大模型当前进度并要求继续搜索或调用完成工具说明原因。如果已达到目标数量，则标记任务完成。

第八步，消息历史压缩机制：当消息总数超过五十条时，保留系统提示词（第一条消息）和最近四十条消息，丢弃中间的历史消息，以控制大模型的上下文窗口用量。

第九步，步骤级错误处理：如果当前步骤出错，将错误信息作为用户消息追加到历史中，让大模型自行调整策略继续执行。

循环结束后，记录最终统计信息（结束时间、总步骤数、结果数量），返回采集到的结果数组。

**取消机制**：设置取消标志和完成标志，主循环在下次检查条件时会退出。

### 5.4 提示词模块（background/agent/prompts.ts）

**日期感知**：提供一个工具函数，获取当前日期的年、月、日、中文格式日期字符串以及英文月份名。这确保了每次运行时，提示词中嵌入的都是实时日期信息。

**系统提示词**：动态生成一段中文提示词，定义 Agent 的角色（专业 AI 资讯搜索 Agent）和任务目标（收集十到二十条高质量资讯）。提示词中详细列举了五大搜索策略：

第一策略是 Twitter/X 平台，推荐访问 Karpathy、Andrew Ng、Yann LeCun 等 AI 领域意见领袖的主页，以及搜索特定话题标签。

第二策略是 GitHub Trending 页面，关注 Python 和 TypeScript 语言的热门项目，特别是 LLM、Agent、ML 相关的。

第三策略是 Google 搜索，使用动态生成的包含当前年月的搜索关键词。

第四策略是 ArXiv 论文平台，关注 AI、计算语言学和机器学习三个分类的最新论文。

第五策略是 Reddit 社区，关注机器学习和本地大模型两个子社区。

提示词还列举了所有可用工具及其用途说明，并给出了操作注意事项：发现即保存、提供详细信息、避免重复、页面加载失败时尝试等待或刷新、某个网站不可访问时切换策略、达到目标数量后完成任务。

**初始用户消息**：生成一条具体的指令，要求 Agent 从 Twitter 上的 Karpathy 主页开始搜索，明确告知当前日期，并指示立刻导航到目标页面。

为了向后兼容，系统提示词同时以常量形式导出。

### 5.5 工具定义与执行模块（background/agent/tools.ts）

**工具定义**：以 OpenAI 兼容的函数调用格式定义了十一个工具，每个工具都有中文描述、参数规格（JSON Schema 格式）和必填参数标记。这些工具分为三类：

浏览器操作类工具——导航到 URL、点击元素（CSS 选择器定位）、在输入框输入文本、按键盘按键、滚动页面（上或下）。

信息获取类工具——获取页面文本内容、获取页面所有链接。

任务管理类工具——等待指定秒数（一到十秒范围限制）、保存搜索结果（包含标题、URL、多个可选链接、创新点描述、公司名称、热度等级、标签数组、来源等丰富字段）、获取当前搜索状态、完成任务（附原因说明）。

**工具执行器**：一个类，持有 Agent 状态的引用。其核心方法是 execute，接收工具名称和参数字典，通过 switch 语句路由到对应的执行逻辑。

浏览器操作类工具的执行委托给浏览器上下文和页面控制器。信息获取类工具会先检查页面是否已连接，未连接则返回中文错误提示。获取页面内容时会同时返回页面标题、URL 和正文前八千字符的摘要。获取链接时限制最多返回前五十个，并统计总数。

等待工具的实现将用户请求的秒数钳制在一到十的范围内，然后执行异步等待。

保存结果工具的执行逻辑最为复杂：首先从参数中构造搜索结果对象，缺失字段使用默认值（如公司默认为"Community"、热度默认为"Medium"、标签默认为包含"AI"的单元素数组）。然后执行去重检查——如果已有结果中存在标题相同（不区分大小写）、或来源 URL 相同、或 GitHub 链接相同的条目，则跳过并返回提示。通过去重后将结果加入状态中的结果数组。如果结果数量达到目标值，则自动标记任务完成。

获取状态工具返回一段格式化的中文文本，列出已保存数量/目标数量、当前步骤/最大步骤、是否完成、以及所有已保存项目的编号列表。

完成工具将任务标记为已完成，返回完成原因和已保存数量。

对于未知工具名称，返回中文的"未知工具"提示。

### 5.6 浏览器上下文管理器（background/browser/context.ts）

这个模块负责管理多个浏览器标签页的连接和切换，采用单例模式。

内部维护两个状态：当前活动标签页的 ID，以及一个以标签页 ID 为键、页面控制器实例为值的映射表。

**获取或创建页面**：查询映射表中是否已有该标签页对应的页面控制器，若有直接返回，若无则创建新的页面控制器实例。

**连接页面**：如果映射表中已有该页面则直接成功，否则调用页面控制器的连接方法，成功后加入映射表。

**获取当前页面**：根据当前标签页 ID 查找映射表，如果找不到则创建并连接。

**切换标签页**：先通过 Chrome API 激活目标标签页，然后获取或创建并连接其页面控制器，更新当前标签页 ID。

**导航到 URL**：如果当前页面已连接，直接使用页面控制器导航。否则通过 Chrome 标签页 API 更新 URL，等待标签页加载完成后重新创建和连接页面控制器。

**等待标签页加载**：监听 Chrome 标签页更新事件，当目标标签页的状态变为"complete"时解除阻塞。同时设置三十秒超时保护，超时后无论是否加载完成都继续执行。

**清理**：断开所有已连接页面的连接，清空映射表，重置当前标签页 ID。

**移除指定页面**：断开指定标签页的连接，从映射表中删除，如果被移除的是当前标签页则重置当前标签页 ID。

### 5.7 页面控制器（background/browser/page.ts）

这个模块封装了对单个浏览器标签页的所有操作，通过 Chrome Extension 的脚本注入 API 来控制页面。

由于使用的是 Chrome 扩展 API 而非 Chrome DevTools Protocol，连接和断开操作都是空操作，attached 属性始终返回 true。

**导航**：通过 Chrome 标签页 API 更新标签页的 URL，然后等待页面加载完成。等待机制与上下文管理器类似——监听标签页更新事件并设置三十秒超时，但在检测到加载完成后额外等待五百毫秒以确保页面渲染稳定。

**获取 URL 和标题**：通过 Chrome 标签页 API 直接查询标签页信息。

**脚本注入执行**：提供一个通用的私有方法，使用 Chrome 脚本注入 API 在目标标签页中执行传入的函数，返回执行结果。错误时静默返回 null。

**点击元素**：向标签页注入脚本，使用 CSS 选择器查找元素，找到后调用其 click 方法。未找到元素时返回中文错误提示。成功点击后额外等待五百毫秒让页面响应。

**输入文本**：向标签页注入脚本，定位输入框元素后设置焦点、直接赋值 value 属性、然后依次触发 input 和 change 事件使框架（如 React）能感知到值的变化。

**按键**：向标签页注入脚本，在当前聚焦元素上派发键盘按下事件。对于 Enter 键，额外检查当前元素是否在表单内，若在则触发表单的 submit 事件。成功后等待五百毫秒。

**滚动**：向标签页注入脚本，以五百像素为单位进行垂直滚动。成功后等待三百毫秒。

**获取页面文本**：注入脚本后先克隆页面 body 元素（避免修改原页面），移除所有 script、style、noscript 标签，然后使用 TreeWalker 遍历所有文本节点，收集非空文本，拼接后截取前一万五千个字符返回。

**获取链接**：注入脚本查找所有带 href 属性的 a 标签，提取每个链接的文本（截取前一百个字符）和 href 属性值，过滤掉文本或链接为空的条目。

**等待选择器**：以五百毫秒为间隔轮询检查指定 CSS 选择器的元素是否出现在页面上，默认超时十秒。

### 5.8 大模型客户端模块（background/llm/dashscope.ts）

此模块封装了与阿里云百炼（DashScope）平台的通信，采用单例模式。

**配置**：从 Vite 构建时注入的环境变量中读取三个配置项——API 密钥、API 基地址（默认指向阿里云百炼的 OpenAI 兼容接口）、模型名称（默认使用 qwen-max）。

**客户端初始化**：构造函数中检查 API 密钥是否存在，不存在则抛出异常并提示用户配置环境变量。使用 openai 库创建客户端实例，设置密钥和基地址，并开启 dangerouslyAllowBrowser 选项（因为 Chrome 扩展实际运行在浏览器环境中，但该库默认禁止浏览器端使用）。

**聊天接口**：接收消息数组和可选的工具列表，调用 OpenAI 兼容 API 的聊天补全端点。当提供了工具列表时，tool_choice 设为 auto，让模型自行决定是否使用工具。返回完整的 API 响应对象。

**简单补全**：接收一个纯文本提示，以 user 角色发送并返回模型生成的文本内容。

### 5.9 弹出窗口 HTML（popup/index.html）

语言标记为中文简体。页面结构包含五个区域：

**顶部区域**：左侧是"FOMO"字样的 Logo，右侧是一个状态指示圆点。

**进度环**：一个 SVG 圆环动画，由背景环和填充环组成。背景环为灰色，填充环为绿色，通过 stroke-dashoffset 控制进度。圆环内部叠加显示当前数字和总数（"X/20"格式）。SVG 视窗为 120x120，圆环半径为 52，周长约 327。

**当前动作**：居中显示一行文字，表示 Agent 当前正在执行的操作。

**控制按钮组**：四个圆形按钮——开始按钮（播放图标）、停止按钮（方块图标）、下载结果按钮（下载箭头图标）、下载日志按钮（文档图标）。所有图标均使用内联 SVG。

**错误提示**：一个默认隐藏的红色消息区域。

### 5.10 弹出窗口样式（popup/popup.css）

整体设计为宽度固定 240 像素、最小高度 280 像素的极简风格。使用 CSS 自定义属性定义了一套配色方案：白色主背景、浅灰次背景、翠绿色强调色（对应十六进制值 10b981）、深灰主文字色、灰色次文字色、红色危险色、琥珀色警告色。

字体使用系统原生字体栈，以 Apple 系统字体优先。

状态指示圆点根据不同状态有不同表现：空闲时为灰色；运行中为绿色并带发光阴影和呼吸脉冲动画（在一点五秒周期内循环缩放和透明度变化）；已完成为绿色；出错为红色。

进度环填充使用圆角端点，带有零点五秒的平滑过渡动画。进度数字使用 32 像素的超细字重。

四个控制按钮均为 36 像素的圆形，带有悬停放大和按下缩小的微交互效果。开始按钮在启用状态下为绿色底白色图标；停止按钮在启用状态下为红色底白色图标；下载按钮在启用状态下为灰底绿色边框绿色图标；日志按钮在启用状态下为灰底琥珀色边框琥珀色图标。禁用状态统一降低至三成透明度并改为禁止光标。

当前动作文字大小为 11 像素，设置最大宽度 160 像素并在溢出时以省略号截断。错误提示使用 11 像素文字、红色文字配红色半透明背景、圆角容器。

### 5.11 弹出窗口交互逻辑（popup/popup.ts）

**DOM 引用**：在脚本加载时立即获取所有 UI 元素的引用——状态指示圆点、进度环 SVG 圆形元素、进度数字、当前动作文本、四个控制按钮、错误消息区域。

**进度环常量**：圆环周长为 327（2 乘以圆周率乘以 52 的近似值），总目标数为 20。

**状态变量**：维护一个结果缓存数组和一个布尔标记表示是否有日志可下载。

**进度环更新**：根据当前进度数计算 stroke-dashoffset 偏移量，公式为周长减去进度比例乘以周长，同时更新数字显示。

**UI 状态更新**：接收状态更新数据后，更新状态圆点的 CSS 类名以切换颜色和动画；更新进度环；更新当前动作文字；根据状态调整四个按钮的禁用/启用状态——运行中时开始按钮禁用、停止按钮启用、下载按钮禁用；已完成时开始按钮启用、停止按钮禁用、两个下载按钮启用；出错时如有错误信息则显示错误区域；空闲时根据是否有缓存结果和日志决定下载按钮的状态。

**下载结果**：将缓存的结果数组转为格式化的 JSON 字符串，创建 Blob 对象和对象 URL，使用 Chrome 下载 API 触发保存对话框。文件名包含时间戳，格式为 fomo-results-时间戳.json。

**下载日志**：向 Background 发送下载日志请求，收到日志对象后同样生成带时间戳的 JSON 文件并触发下载。文件名格式为 fomo-logs-时间戳.json。

**启动任务**：先查询当前活动标签页，检查标签页 ID 是否存在且 URL 不是 Chrome 内部页面（以 chrome:// 或 chrome-extension:// 开头的页面不允许操作）。清空缓存和进度环，向 Background 发送启动消息，同时将 UI 切换到运行状态。

**停止任务**：向 Background 发送停止消息，UI 回到空闲状态，动作文本显示"Stopped"。

**消息监听**：注册监听器接收 Background 推送的三类消息——状态更新（直接刷新 UI）、任务完成（缓存结果并更新 UI）、错误（更新 UI 并显示错误）。

**初始化**：脚本加载后立即向 Background 发送获取状态的消息（用于 Popup 在任务进行中被重新打开的场景），并为四个按钮绑定点击事件。

## 六、模块间数据流

整体数据流分为两条路径：

**控制流（Popup → Background）**：用户点击开始按钮 → Popup 获取当前标签页 ID → 发送 START_TASK 消息到 Background → Background 初始化浏览器上下文和 Agent 执行器 → 执行器开始 ReAct 循环。用户点击停止按钮 → 发送 STOP_TASK → 执行器被取消。用户点击下载按钮 → 发送 DOWNLOAD_RESULTS 或 DOWNLOAD_LOGS → Background 同步返回数据 → Popup 生成文件触发下载。

**状态流（Background → Popup）**：Agent 执行器通过状态回调函数 → Background 入口模块接收 → 通过 Chrome 消息 API 推送 STATUS_UPDATE → Popup 消息监听器接收 → 更新 UI。任务完成时推送 TASK_COMPLETE（附带完整结果数组）。出错时推送 ERROR（附带错误消息字符串）。

**Agent 内部数据流**：执行器维护完整的消息历史列表，每轮循环中将消息列表发送给大模型 → 大模型返回思考内容和/或工具调用指令 → 执行器将大模型响应追加到消息历史 → 如有工具调用则委托给工具执行器 → 工具执行器根据工具类型将浏览器操作委托给浏览器上下文和页面控制器 → 页面控制器通过 Chrome 脚本注入 API 在目标标签页中执行 JavaScript → 执行结果返回给工具执行器 → 工具结果追加到消息历史 → 进入下一轮循环。

保存结果工具被调用时，数据流为：大模型生成工具调用参数 → 工具执行器构造搜索结果对象并执行去重 → 结果存入 Agent 状态的结果数组 → 通过状态回调通知进度变化。

## 七、关键业务逻辑说明

### ReAct 循环控制

Agent 的核心工作模式是"思考-行动-观察"的迭代循环。大模型既是"大脑"也是"决策者"，它根据系统提示词理解任务目标，根据当前消息历史中积累的上下文（已执行的操作、已获取的页面内容、已保存的结果）来规划下一步行动。大模型可以自由组合各种工具来完成任务——先导航到某个网站，获取页面内容分析信息，发现有价值的项目就保存结果，然后决定继续在当前页面查找还是切换到其他数据源。

### 任务完成判定

任务有三种完成条件：一是保存的结果数量达到目标值（默认 20 条）时工具执行器自动标记完成；二是大模型在足够结果的情况下自行选择停止对话；三是步骤数达到上限（默认 100 步）循环自然退出。如果大模型过早停止但结果数量不足，系统会自动注入一条催促消息要求继续搜索。

### 去重策略

通过三重比对实现去重：标题不区分大小写的完全匹配、来源 URL 完全匹配、GitHub 链接完全匹配。任一条件匹配即视为重复并跳过。

### 消息历史压缩

当消息总数超过五十条时，仅保留系统提示词和最近四十条消息。这个策略在保证大模型有足够上下文理解当前状态的同时，防止因上下文过长导致 API 调用失败或成本飙升。

### 页面操作的鲁棒性

所有页面操作都包裹在错误处理中，失败时返回描述性的中文错误信息而非抛出异常。导航操作有三十秒的超时保护。点击和按键操作后有短暂等待以让页面响应。页面文本提取通过克隆 DOM 树避免影响原页面的正常运行。

## 八、错误处理策略

项目的错误处理采用"优雅降级、自我修复"的策略：

**Agent 级错误恢复**：每个步骤的执行都被 try-catch 包裹。当某一步出错时，错误信息被转化为用户消息注入到对话历史中，让大模型自行理解错误原因并调整策略（如换一个网站访问、尝试不同的选择器等），而非直接终止整个任务。

**消息推送的静默失败**：所有从 Background 向 Popup 推送消息的操作都捕获并忽略异常，因为 Popup 窗口可能未打开。

**API 密钥校验**：大模型客户端在构造时检查密钥是否配置，若缺失则抛出包含配置指引的错误消息。

**页面操作的防御编程**：每个需要页面的工具调用都先检查页面是否连接，未连接时返回中文错误提示而非崩溃。脚本注入失败时静默返回空值或空数组。

**标签页生命周期管理**：监听标签页关闭事件，及时清理对应的页面控制器，防止内存泄漏和无效操作。

**工具参数解析**：JSON 解析工具调用参数时，解析失败使用空对象兜底，防止因大模型输出格式问题导致执行中断。

**任务级错误状态**：任务执行的整体 try-catch 捕获未预期的异常，将状态标记为 error 并向 Popup 发送错误通知。finally 块确保浏览器上下文被清理、执行器引用被释放。
