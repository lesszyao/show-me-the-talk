# FOMO — AI 热点资讯自动搜索 Chrome 浏览器扩展 · 项目描述

---

## 一、项目目的与高层架构

FOMO 是一个 Chrome 浏览器扩展（插件），其核心用途是：**一键在互联网上自动搜索并收集当前最新、最热门的人工智能、大语言模型、智能体、代码助手等领域的开源项目和学术论文**。用户点击扩展图标后，扩展会在当前标签页中由一个"AI 智能体"自动驾驶浏览器——访问 Twitter/X、GitHub Trending、ArXiv、Reddit、Google 等网站——像人一样浏览、滚动、点击、阅读页面内容，将发现的高价值资讯逐条保存，最终生成一份结构化的 JSON 结果供用户下载。

从架构上看，项目分为三个核心层：

- **前端弹出面板层（Popup）**：用户点击扩展图标后显示的小窗口界面，提供启动、停止、下载等操作按钮，以及实时进度展示。
- **后台服务工作线程层（Background Service Worker）**：Chrome 扩展的后台常驻逻辑，充当 Popup 与 Agent 执行器之间的调度器和消息中枢。
- **AI 智能体层（Agent）**：包含 LLM 调用客户端、提示词模板、工具定义与执行器、浏览器页面控制器，构成一个完整的 ReAct（推理-行动）循环引擎。

数据流方向为：用户通过 Popup 发起指令 → Background 接收指令并创建 Agent 执行器 → Agent 循环调用 LLM 获取决策 → LLM 返回工具调用指令 → 工具执行器操控浏览器标签页进行实际网页交互 → 交互结果反馈给 LLM → LLM 继续推理直至收集足够结果 → 结果返回 Background → Background 通知 Popup 更新界面。

## 二、技术栈与构建配置

### 编程语言

项目使用 TypeScript 编写全部源码。TypeScript 配置采用严格模式，目标编译版本为 ES2022，模块系统为 ESNext，模块解析策略为 bundler 模式。类型声明依赖了 Chrome 扩展 API 类型定义、Vite 客户端类型和 Node 类型三种类型包。项目配置不产出声明文件，仅做类型检查。

### 构建工具

使用 Vite 5.x 作为构建和打包工具。Vite 配置了多入口构建：一个入口是弹出面板的 HTML 文件，另一个入口是后台服务工作线程的 TypeScript 文件。构建产出目录为项目根目录下的 dist 文件夹，每次构建前会清空该目录。构建目标为 esnext，且刻意关闭了代码压缩（方便调试），同时开启了 sourcemap 生成。

Vite 的 Rollup 配置中，将 chrome 全局对象和所有 Node 内置模块（以 node: 前缀开头的）标记为外部依赖，不打包进产物。输出文件的命名规则是：后台脚本固定输出为 background 目录下的 index.js；其他入口按"入口名/入口名.js"的格式输出；代码分割产生的公共 chunk 放入 chunks 目录并带哈希后缀；CSS 资源固定输出为 popup 目录下的 popup.css。

构建配置中还包含一个自定义 Vite 插件，名为 copy-manifest，它在打包完成后执行以下操作：将扩展清单文件复制到构建产出目录；将弹出面板的 CSS 文件也复制过去；将 icons 目录下的所有 PNG 和 SVG 图标文件复制到产出目录的 icons 子目录中。

Vite 还通过环境变量注入机制，在构建时将三个环境变量硬编码到产物中：DashScope API 密钥、DashScope 基础 URL 地址、以及所使用的模型名称。这些变量通过 Vite 的 loadEnv 函数从项目根目录的环境变量文件中读取。

## 五、各模块职责与接口

### 5.1 全局类型定义模块（types.ts）

该文件是整个项目的类型契约中心，定义了六组核心数据结构：

- **日志条目结构**：包含时间戳、日志类型（助手消息、工具调用、工具响应、用户消息、系统事件、错误共六种）、步骤编号、以及任意类型的内容字段。
- **执行日志结构**：包含任务的起止时间、总步骤数、结果数量、以及日志条目数组。用于完整记录一次 Agent 运行的全过程。
- **搜索结果结构**：每条搜索结果包含名称、信息来源 URL、可选的产品 URL、可选的 GitHub 链接、可选的 ArXiv 链接、创新点描述、所属公司或社区名称、热度趋势（高/中/低）、标签数组、来源平台名称、发现时间戳。
- **Agent 状态结构**：包含已收集的结果数组、已访问 URL 集合（用于去重）、当前步骤号、最大步骤数限制、目标收集数量、以及是否完成的布尔标志。
- **执行状态枚举**：包含五种状态——空闲、运行中、暂停、已完成、错误。
- **消息类型联合**：定义了 Popup 与 Background 之间通信的所有消息类型，包括启动任务（携带标签页 ID）、停止任务、获取状态、下载结果、下载日志、状态更新推送、任务完成通知、错误通知。

### 5.2 弹出面板模块（popup 目录）

#### HTML 入口文件

页面语言设置为中文。整体布局为一个垂直居中的容器，从上到下依次包含：顶部状态栏（扩展名称 LOGO 加一个状态指示圆点）、SVG 环形进度条（默认显示 0/20 的进度）、当前操作文字提示区、操作按钮组、错误消息提示区。按钮组包含四个圆形按钮：播放（启动搜索）、停止、下载结果、下载日志。停止和两个下载按钮初始状态为禁用。HTML 文件末尾通过 module 类型的 script 标签引入 popup.ts 逻辑脚本。

#### CSS 样式文件

采用 CSS 自定义属性（变量）定义主题色系：主背景色为白色，次背景色为浅灰色，主题强调色为翡翠绿色，文字主色为深灰近黑色，文字次色为中灰色，错误色为红色，警告色为琥珀色。

弹出面板固定宽度为 240 像素，最小高度 280 像素。采用 Apple 系统字体栈。状态指示圆点尺寸为 8 像素，在运行状态下会以翡翠绿色发光脉冲动画呈现（1.5 秒循环）。进度环使用 SVG 实现，外环半径 52 像素，周长约 327 像素，通过动态修改 stroke-dashoffset 属性来表现进度。按钮均为 36 像素的圆形，具有悬浮放大和按压缩小的微交互效果。禁用状态的按钮透明度降至 30%。错误消息区有半透明红色背景和圆角样式，通过 hidden 类控制显示隐藏。

#### TypeScript 逻辑文件

该模块在脚本加载时立即获取所有需要操作的 DOM 元素引用。定义了进度环周长常量（327）和搜索目标总量常量（20）。维护两个模块级变量：缓存的搜索结果数组和是否有日志的布尔标志。

核心功能包括：

- **进度环更新函数**：根据当前进度数除以总量计算 stroke-dashoffset 偏移值，更新 SVG 圆环的视觉进度和中心数字文本。
- **UI 状态更新函数**：接收一个状态更新数据对象，根据执行状态（运行中、已完成、出错、空闲）分别设置状态圆点的样式类、更新进度环、切换按钮的启用/禁用状态、显示当前操作文案或错误信息。
- **结果下载函数**：将缓存的搜索结果序列化为格式化的 JSON 字符串，构造 Blob 对象和 Object URL，调用 Chrome 下载 API 以"fomo-results-时间戳.json"为文件名触发下载（弹出保存对话框）。
- **日志下载函数**：向 Background 发送下载日志的消息请求，收到响应后同样构造 JSON Blob 触发下载，文件名格式为"fomo-logs-时间戳.json"。
- **启动任务函数**：首先通过 Chrome Tabs API 查询当前活动标签页，校验该标签页不是 Chrome 内部页面（chrome:// 或 chrome-extension:// 开头的 URL 会被拒绝），然后清空之前的缓存结果和进度，向 Background 发送启动任务消息（携带标签页 ID），并立即将 UI 切换到运行中状态。
- **停止任务函数**：向 Background 发送停止消息，将 UI 恢复到空闲状态。

模块通过 Chrome Runtime 消息监听器接收 Background 推送的三种消息：状态更新（更新进度 UI）、任务完成（缓存结果并切换到完成状态）、错误（显示错误信息）。

初始化时立即向 Background 查询一次当前状态，并为四个按钮绑定点击事件处理函数。

### 5.3 后台入口模块（background/index.ts）

这是 Chrome 扩展 Service Worker 的入口文件。维护四个模块级状态变量：当前执行器实例（可为空）、当前搜索结果数组、当前执行状态、当前执行日志。

提供三个消息发送辅助函数，分别用于向 Popup 推送状态更新、任务完成通知、错误通知。这三个函数都通过 Chrome Runtime 消息 API 发送，并且捕获发送失败的异常静默忽略（因为 Popup 可能未打开）。

**启动任务流程**：首先检查是否已有运行中的任务（防止重复启动）；获取浏览器上下文单例并设置当前标签页 ID；将标签页连接到浏览器上下文（attach 操作）；创建一个 Agent 执行器实例，传入目标收集数量 20 和最大步骤数 100 作为参数；注册状态回调函数使执行器的进度更新能实时推送到 Popup；调用执行器的 run 方法开始 Agent 主循环；循环结束后保存执行日志；发送任务完成消息。无论成功或失败，最终都会清理浏览器上下文的所有连接并置空执行器引用。

**停止任务流程**：调用执行器的取消方法，置空执行器引用，重置状态为空闲，清理浏览器上下文。

**消息监听器**：通过 Chrome Runtime 的 onMessage 监听器处理来自 Popup 的五种消息：启动任务（异步执行）、停止任务、获取状态（返回当前进度和状态）、下载结果（返回缓存的结果数组）、下载日志（返回执行日志对象）。对未知消息类型返回错误提示。所有消息处理器都返回 true 以保持消息通道异步开放。

还注册了两个 Chrome 事件监听器：标签页关闭时从浏览器上下文中移除已连接的页面；扩展安装或更新时输出日志。

### 5.4 Agent 执行器模块（agent/executor.ts）

这是整个项目的核心引擎，实现了经典的 ReAct（推理-行动）循环模式。

**构造函数**：初始化 Agent 状态对象（结果数组为空、已访问 URL 为空集合、当前步骤为 0、默认目标 20 条、默认最大 100 步）；构建初始消息历史（包含系统提示词和首条用户消息）；创建工具执行器实例（共享状态引用）；初始化执行日志。

**主循环逻辑（run 方法）**：获取 DashScope LLM 客户端单例后进入 while 循环。循环终止条件有三个：状态被标记为完成、被用户取消、或者当前步骤达到最大步骤数限制。每次循环：

1. 将当前消息历史发送给 LLM，同时传入工具定义列表，让 LLM 自动决定下一步行动。
2. 解析 LLM 返回的响应，提取助手消息。如果响应为空则跳过本轮。
3. 如果助手消息中包含文本内容，将前 100 个字符作为当前操作提示推送给 Popup。
4. 将完整的助手消息（包括文本和工具调用）添加到消息历史中。
5. 如果助手消息包含工具调用，则遍历每一个工具调用：解析 JSON 格式的参数（解析失败则使用空对象）；调用工具执行器执行该工具并获得结果字符串；将工具执行结果作为 tool 角色消息添加到历史中。
6. 如果 LLM 返回的完成原因是"stop"且没有工具调用（表示 LLM 认为对话结束），则检查是否已达到目标数量：若未达到，则自动注入一条用户消息督促 LLM 继续搜索或调用 finish 工具说明原因；若已达到，则标记任务完成。
7. 消息历史管理：当消息数量超过 50 条时，保留第一条系统提示词和最近 40 条消息，丢弃中间的历史消息以避免超出 LLM 上下文窗口。

**错误处理**：循环中的每一步都被 try-catch 包裹。捕获到错误后，将错误信息作为一条用户消息注入到消息历史中，提示 LLM"出现了错误，请尝试其他方法继续搜索"，从而让循环继续而不是中断。

**日志记录**：循环中的每个关键事件（助手消息、工具调用、工具响应、系统事件、错误）都会被记录到执行日志中，包含时间戳、类型、步骤号和内容。工具响应内容超过 5000 字符时会被截断。

### 5.5 提示词模块（agent/prompts.ts）

该模块负责生成两份动态文本：系统提示词和首条用户消息。

**日期信息函数**：每次调用时获取当前实际日期，返回年、月、日、中文格式日期字符串、英文月份名称。这确保了提示词中的时间始终是最新的。

**系统提示词内容**：定义了 Agent 的角色身份——"专业的 AI 资讯搜索 Agent"。明确了当前日期（动态注入），要求搜索当前年月最新的内容。设定了目标为收集 10 到 20 条高质量的 AI 相关资讯，涵盖三个类别：热门开源项目、重要学术论文、以及 AI 领域意见领袖的推荐。

提供了五种搜索策略供 Agent 选择和组合：第一优先级是 Twitter/X 平台，列出了 Karpathy、Andrew Ng、Yann LeCun 三位 AI 领域大咖的个人主页链接以及建议搜索的话题标签；第二策略是 GitHub Trending 页面，建议筛选 Python 和 TypeScript 语言的 LLM/Agent/ML 相关项目；第三策略是 Google 搜索，提供了多个包含当前年月的建议搜索关键词模板；第四策略是 ArXiv 学术论文网站，指明了 AI、计算语言学和机器学习三个分类目录；第五策略是 Reddit 社区的机器学习和本地大语言模型两个子版。

提供了全部 11 个可用工具的使用指南（仅以工具名和功能简述的形式）。提出了六条注意事项：发现即保存、提供详细信息、避免重复、页面加载问题时等待或刷新、网站无法访问时切换策略、达到 20 条后调用完成工具。

模块同时导出了常量形式的系统提示词（在模块加载时生成一次）和函数形式的系统提示词（每次调用时重新生成）以兼容不同使用场景。

**首条用户消息**：指示 Agent 立即从访问 Karpathy 的 Twitter 主页开始搜索，明确目标为 20 条不重复的高质量 AI 资讯，并提醒当前的精确日期。

### 5.6 工具定义与执行器模块（agent/tools.ts）

该模块包含两部分：工具的 JSON Schema 定义（供 LLM 理解可用工具）和工具的实际执行逻辑。

**工具定义**：以 OpenAI SDK 的 ChatCompletionTool 格式定义了 11 个工具函数：

- **navigate**：接受一个 URL 字符串参数，用于导航到指定网页。
- **click**：接受一个 CSS 选择器参数，用于点击页面上的指定元素。
- **type_text**：接受 CSS 选择器和文本两个参数，用于在输入框中输入文字。
- **press_key**：接受一个按键名称参数，用于模拟键盘按键。
- **scroll**：接受一个方向参数（上或下），用于滚动页面。
- **get_page_content**：无参数，用于获取当前页面的文本内容。
- **get_links**：无参数，用于获取页面上的所有链接。
- **wait**：接受秒数参数（1 到 10 之间），用于等待页面加载。
- **save_result**：接受多个字段参数（名称、URL、产品 URL、GitHub 链接、ArXiv 链接、创新点、公司、热度、标签、来源），其中名称、URL、创新点和来源为必填，其余为可选。用于保存发现的资讯。
- **get_status**：无参数，用于查询当前搜索进度。
- **finish**：接受一个原因字符串参数，用于标记任务完成。

**工具执行器类**：持有 Agent 状态的引用。提供一个统一的执行入口方法，根据工具名称进行分派。

- navigate 工具：如果当前有已连接的页面对象，通过页面对象导航；否则通过浏览器上下文导航（会创建新的页面连接）。
- click、type_text、press_key、scroll 工具：均要求页面已连接，否则返回错误消息；正常情况下委托给页面控制器的对应方法。
- get_page_content 工具：获取页面文本内容后，同时获取当前 URL 和标题，拼接成"当前页面标题 + URL + 内容摘要"的格式返回，内容摘要截取前 8000 个字符。
- get_links 工具：获取页面所有链接后，只返回前 50 个，格式为"链接文本: 链接地址"的列表。
- wait 工具：参数值被钳制在 1 到 10 秒之间，使用 Promise 和 setTimeout 实现延时等待。
- save_result 工具：构造搜索结果对象（为可选字段提供默认值：公司默认"Community"、热度默认"Medium"、标签默认"AI"、来源默认"Unknown"），然后进行去重检查——如果结果的名称（不区分大小写）、URL 或 GitHub 链接与已有结果重复，则跳过。通过检查后将结果推入状态中的结果数组。如果数组长度达到目标数量，自动标记任务完成。
- get_status 工具：返回格式化的状态文本，包含已保存结果数/目标数、当前步骤/最大步骤、是否完成、以及已保存的所有项目列表。
- finish 工具：标记 Agent 状态为完成，返回结束原因和已保存的结果数量。
- 遇到未知工具名时返回提示信息。

### 5.7 浏览器上下文管理模块（browser/context.ts）

该模块采用单例模式管理浏览器中多个标签页的连接生命周期。

内部维护两个状态：当前活跃标签页 ID（可为空）和一个以标签页 ID 为键、页面控制器对象为值的 Map 缓存。

主要方法包括：

- **获取或创建页面**：查找 Map 中是否已有该标签页的页面对象，有则返回，没有则创建新实例。
- **连接页面**：调用页面对象的 attach 方法建立连接，成功后存入 Map 缓存。
- **获取当前页面**：根据当前标签页 ID 从缓存中查找，如果缓存未命中则自动创建并连接。
- **切换标签页**：通过 Chrome Tabs API 激活目标标签页，然后获取或创建该标签页的页面对象。
- **导航到 URL**：如果当前页面已连接则委托页面对象导航；如果页面未连接则使用 Chrome Tabs API 更新标签页 URL，等待加载完成后重新创建页面连接。
- **等待标签页加载**：通过监听 Chrome Tabs 的 onUpdated 事件，等待目标标签页的状态变为"complete"。设有 30 秒超时保护。
- **清理**：遍历所有已连接的页面执行 detach 操作，清空 Map 缓存和当前标签页 ID。
- **移除指定页面**：从 Map 中删除指定标签页并执行 detach，如果该标签页正好是当前标签页则同时置空。

模块底部通过闭包实现单例导出函数：第一次调用时创建实例，后续调用返回同一实例。

### 5.8 页面控制器模块（browser/page.ts）

该模块封装了对单个 Chrome 标签页的所有 DOM 操作能力。使用 Chrome Extension 的 scripting API（而非 Chrome DevTools Protocol）来注入脚本到目标标签页中执行。

**连接与断开**：由于 Chrome scripting API 不需要显式的 attach/detach 连接过程，因此 attach 方法直接返回成功，detach 方法为空操作，attached 属性始终为真。这些方法的保留是为了与浏览器上下文管理器的接口保持一致。

**页面导航**：调用 Chrome Tabs API 更新标签页 URL，然后通过监听 onUpdated 事件等待页面加载完成（状态变为 complete），额外等待 500 毫秒确保页面渲染稳定。设有 30 秒超时保护以防页面永远不加载完成。

**脚本执行基础方法**：封装了 Chrome scripting.executeScript API 的通用调用模式，将一个函数注入到目标标签页中执行并返回结果。错误时返回空值并输出控制台日志。

**点击操作**：注入脚本在目标页面中通过 CSS 选择器查找元素，找不到则返回错误信息；找到则调用元素的 click 方法。点击后等待 500 毫秒等待页面响应。

**文本输入操作**：注入脚本查找输入元素，先聚焦（focus），然后直接设置 value 属性，并手动触发 input 和 change 两个冒泡事件，以确保前端框架（如 React、Vue）能够检测到值的变化。

**按键操作**：注入脚本在当前焦点元素上触发 keydown 事件。对于 Enter 键有特殊处理——会查找最近的表单父元素并触发其 submit 事件。按键后等待 500 毫秒。

**页面滚动**：注入脚本调用 window.scrollBy 方法，向下滚动 500 像素或向上滚动 500 像素。滚动后等待 300 毫秒。

**获取页面文本**：注入一段较复杂的脚本——先克隆 body 元素（避免修改原页面），移除其中所有 script、style、noscript 标签，然后使用 TreeWalker 遍历所有文本节点，收集非空文本内容，用空格拼接后截取前 15000 个字符返回。

**获取页面链接**：注入脚本查找所有带有 href 属性的 a 标签，提取链接文本（截取前 100 字符）和 href 值，过滤掉文本或链接为空的条目。

**等待选择器出现**：轮询机制，每 500 毫秒检查一次目标 CSS 选择器对应的元素是否存在于 DOM 中，默认超时时间为 10 秒。

### 5.9 LLM 客户端模块（llm/dashscope.ts）

该模块封装了与阿里云百炼（DashScope）大语言模型 API 的通信。

**配置读取**：从 Vite 构建时注入的环境变量中读取三个配置项——API 密钥、基础 URL（默认为阿里云 DashScope 兼容模式 API 地址）、模型名称（默认为 qwen-max）。

**客户端初始化**：构造函数中首先检查 API 密钥是否存在，不存在则抛出错误并提示用户在 .env.local 文件中设置。然后使用 OpenAI SDK 创建客户端实例，传入 API 密钥和基础 URL。有一个关键配置是 dangerouslyAllowBrowser 设为真——因为 OpenAI SDK 默认禁止在浏览器环境中直接调用 API（出于密钥安全考虑），但 Chrome 扩展的 Service Worker 本质上运行在浏览器环境中，因此必须显式开启此选项。

**聊天接口**：接受消息历史数组和可选的工具定义数组，调用 OpenAI SDK 的 chat.completions.create 方法。当传入了工具定义时，tool_choice 设为"auto"（让模型自行决定是否调用工具）；未传入工具时 tool_choice 为未定义。

**简单补全接口**：接受一个纯文本提示，包装为单条用户消息发送给模型，返回模型回复的文本内容。

模块同样采用单例模式：通过闭包变量缓存客户端实例，首次调用时创建，后续复用。

## 六、模块间数据流

整个系统的数据流可以划分为三个通信链路：

**链路一：用户操作 → 任务启动**

用户点击 Popup 的播放按钮 → Popup 脚本通过 Chrome Runtime 消息 API 向 Background 发送 START_TASK 消息（携带当前标签页 ID）→ Background 的消息监听器接收消息 → 调用启动任务函数 → 创建浏览器上下文、连接标签页、实例化 Agent 执行器 → 开始 ReAct 循环。

**链路二：Agent 循环中的 LLM-工具交互**

Agent 执行器将当前消息历史发送到 DashScope API → API 返回助手消息（可能包含工具调用）→ 执行器解析工具调用参数 → 调用工具执行器的 execute 方法 → 工具执行器根据工具名分派到对应处理函数 → 浏览器操作类工具（navigate、click 等）通过浏览器上下文获取当前页面控制器 → 页面控制器通过 Chrome scripting API 向标签页注入脚本并执行 → 执行结果返回给工具执行器 → 工具执行器返回结果字符串给 Agent 执行器 → Agent 执行器将结果作为 tool 角色消息添加到消息历史 → 进入下一轮 LLM 调用。

**链路三：进度推送 → UI 更新**

Agent 执行器在每个关键步骤（开始、工具执行、LLM 思考、保存结果、完成）通过状态回调函数 → 回调函数在 Background 入口中被定义为发送 Chrome Runtime 消息 → 消息到达 Popup 的消息监听器 → 更新进度环、状态圆点、当前操作文案、按钮状态。

**数据存储方面**：搜索结果暂存在 Background Service Worker 的模块级变量和 Agent 状态对象中，不使用持久化存储。用户通过下载按钮导出后以 JSON 文件形式保存到本地磁盘。执行日志同理。

## 七、关键业务逻辑描述

### ReAct 循环控制

Agent 执行器的核心逻辑是一个典型的 ReAct（Reasoning + Acting）循环：LLM 先进行推理思考（输出文本内容），然后决定执行一个或多个工具调用（行动），工具执行后将观察结果反馈给 LLM，LLM 基于新的观察继续推理和行动。这个循环持续运行直到满足终止条件。

### 终止条件判断

循环终止有四种情形：第一，LLM 主动调用 finish 工具声明完成，此时 isComplete 标志被置为真；第二，save_result 工具保存结果后发现数量达到目标值，自动将 isComplete 置为真；第三，循环步骤数达到最大限制（默认 100 步）；第四，用户手动点击停止按钮，通过 cancel 方法将 cancelled 标志置为真。

### LLM 提前结束的补救机制

如果 LLM 在未收集够目标数量结果的情况下就停止了工具调用（finish_reason 为 stop 且无工具调用），执行器不会直接结束循环，而是自动注入一条用户消息提醒 LLM 当前进度不足，要求继续搜索或明确说明无法继续的原因。

### 消息历史压缩策略

大语言模型有上下文窗口长度限制。当消息历史超过 50 条时，执行器会保留第一条（系统提示词，确保角色设定不丢失）和最后 40 条消息，丢弃中间部分。这是一种简单但有效的滑动窗口策略。

### 搜索结果去重算法

保存结果时进行三重去重检查：按名称（转小写后精确匹配）、按来源 URL（精确匹配，需非空）、按 GitHub URL（精确匹配，需非空）。任一维度命中即判定为重复并跳过。

### 标签页连接生命周期管理

浏览器上下文通过 Map 缓存维护标签页到页面控制器的映射关系。当标签页被用户关闭时，Chrome 的 tabs.onRemoved 事件触发清理。任务完成或失败时，cleanup 方法批量断开所有连接并清空缓存。

### 页面加载等待机制

导航到新 URL 后，系统通过监听 Chrome Tabs 的 onUpdated 事件来检测页面加载状态。当状态变为"complete"时认为加载完成，之后额外等待 500 毫秒让页面渲染稳定。所有等待操作都设有 30 秒超时保护，防止因页面永远加载不完而导致系统挂起。

## 十、错误处理策略

### Agent 循环层面

每一步 LLM 调用和工具执行都被 try-catch 包裹。捕获到任何错误后，不会中断循环，而是将错误信息作为用户消息注入到对话历史中，提示 LLM 尝试其他方法继续。同时将错误记录到执行日志中。

### 工具执行层面

每个页面操作工具在注入脚本执行时都有独立的 try-catch 错误处理。DOM 元素找不到时返回中文错误提示（如"未找到元素: 选择器"）。脚本执行本身的异常也会被捕获并返回为友好的错误消息字符串。所有工具方法都不会抛出异常给调用方，而是通过返回值传递错误信息。

### 消息通信层面

Background 向 Popup 发送消息时，所有的 Chrome Runtime sendMessage 调用都附加了 catch 子句来静默忽略发送失败的错误——这是必要的，因为用户可能在任务运行期间关闭了 Popup 窗口。

### 前端层面

Popup 的任务启动函数对标签页查询结果进行了空值检查（tab 或 tab.id 不存在时抛出错误），并且检查当前标签页 URL 是否为 Chrome 内部页面（这些页面不允许注入脚本），对这两种异常情况向用户显示错误提示。

### LLM 客户端层面

API 密钥缺失时在构造阶段即抛出明确的错误信息，告知用户需要在环境变量文件中配置。LLM 返回空响应时记录日志并跳过当前步骤（continue），不影响循环继续。

### 资源清理

Background 入口的启动任务函数使用 try-finally 模式确保无论任务成功或失败，都会执行浏览器上下文的清理操作（断开所有标签页连接）和执行器引用置空。

## 十一、边界情况与异常处理

- **重复启动保护**：如果任务已在运行中，再次收到启动请求时直接返回，不会创建新的执行器实例。
- **Chrome 内部页面限制**：用户在 chrome:// 或 chrome-extension:// 页面上点击启动时，会收到"Cannot run on this page"的错误提示。
- **LLM 空响应处理**：如果 API 返回的 choices 数组为空或第一个 choice 不存在，记录日志并跳过当前循环迭代。
- **工具参数解析失败**：JSON.parse 工具调用参数失败时不抛出异常，而是使用空对象作为默认参数，让工具函数自行处理缺失参数的情况。
- **页面未连接时的工具调用**：除 navigate 和 wait 外的浏览器操作工具在页面对象为空时返回"错误: 页面未连接"的提示。
- **链接提取失败**：getLinks 方法在 scripting.executeScript 抛出异常时返回空数组，不影响流程。
- **页面文本提取中的 DOM 清理**：获取页面文本时克隆 body 而不是直接操作原始 DOM，避免影响用户正在浏览的页面内容。
- **标签页关闭事件处理**：如果用户手动关闭了 Agent 正在操作的标签页，onRemoved 事件会自动清理对应的页面连接。如果被关闭的标签页正好是当前标签页，则当前标签页 ID 也会被置空。
- **消息通道保活**：所有 Chrome Runtime 消息监听器的处理函数都返回 true，这是 Chrome 扩展异步消息通信的要求——告知浏览器消息通道需要保持开放直到 sendResponse 被调用。
- **wait 工具的参数钳制**：用户传入的秒数被限制在 1 到 10 之间，防止 LLM 请求过长或过短（甚至负值）的等待时间。
