# FOMO — AI 热点资讯自动搜索 Chrome 浏览器扩展

## 一、项目目的与高层架构

FOMO 是一个 Chrome 浏览器扩展程序，其核心目的是让用户一键启动一个"AI 智能体"（Agent），该智能体会自动操控用户当前的浏览器标签页，在互联网上浏览多个信息源（Twitter/X、GitHub Trending、Google 搜索、ArXiv、Reddit 等），搜集当下最热门的 AI、大语言模型（LLM）、智能体（Agent）、代码助手等方向的开源项目和学术论文信息，最终将结果汇总为结构化数据供用户下载。

整个系统的高层架构可以分为三个层面：

第一层是**用户界面层**（Popup），即用户点击浏览器工具栏图标后弹出的小窗口。这个窗口极简，核心就是一个环形进度条和几个操作按钮，负责展示任务状态、启动/停止任务、下载结果和日志。

第二层是**后台服务层**（Background Service Worker），它是整个扩展程序的运行中枢。当用户在 Popup 中点击"开始"按钮后，后台服务会初始化浏览器上下文、创建智能体执行器，然后启动一个基于 ReAct（Reasoning + Acting）模式的循环。在这个循环中，大语言模型负责"思考"和"决策"，决定下一步要执行什么操作（比如访问某个网站、滚动页面、点击链接、提取内容），然后由工具执行器在真实的浏览器标签页中执行这些操作，将操作结果反馈给大语言模型，形成闭环。

第三层是**大语言模型服务层**（LLM），项目通过阿里云百炼平台（DashScope）提供的与 OpenAI 兼容的接口，调用通义千问系列模型（默认使用 qwen-max）来驱动智能体的推理和决策。

三者之间的通信方式是：Popup 和 Background 之间通过 Chrome 扩展的消息传递机制（runtime.sendMessage / onMessage）进行双向通信；Background 和 LLM 之间通过 HTTP 请求（OpenAI 兼容的聊天补全接口）通信；Background 和浏览器标签页之间通过 Chrome 的脚本注入接口（scripting.executeScript）进行页面操控。

## 二、技术栈与构建配置

### 编程语言

项目全部使用 TypeScript 编写，编译目标为 ES2022。TypeScript 配置启用了严格模式，包括严格的空值检查、严格的函数类型等。模块解析策略采用 bundler 模式（适配 Vite），支持 ESNext 模块语法。类型声明中包含了 DOM、DOM.Iterable 以及 ES2022 的标准库，并额外引入了 chrome 类型定义（来自 @types/chrome）、vite/client 类型和 node 类型。TypeScript 配置中设置了 noEmit 为 true，因为编译工作完全交由 Vite 和 Rollup 处理，TypeScript 仅用于类型检查。

### 构建工具

使用 Vite 5.x 作为构建工具。Vite 配置定义了两个入口点：一个是 Popup 页面的 HTML 文件，另一个是后台脚本的 TypeScript 入口。构建输出目录为 dist，每次构建前会清空该目录。

Rollup 的输出配置中有一套自定义的文件命名规则：后台脚本固定输出为 background/index.js，其他入口按"名称/名称.js"的模式输出，共享代码块输出到 chunks 目录下并附带哈希值，CSS 资源固定输出为 popup/popup.css。

构建目标为 esnext，关闭了代码压缩以便于调试，同时开启了 sourcemap 生成。

chrome 全局对象和所有以 node: 开头的内置模块被标记为外部依赖，不会被打包进产物。

Vite 配置中定义了一个路径别名：@ 符号映射到项目根目录。模块解析优先使用 browser 条件导出，其次是 module 和 main 字段。

### 环境变量

通过 Vite 的 define 配置，三个环境变量会在构建时被静态替换注入到代码中：大语言模型的 API 密钥（VITE_DASHSCOPE_API_KEY）、API 基础地址（VITE_DASHSCOPE_BASE_URL，默认值为阿里云百炼的兼容模式端点）、以及模型名称（VITE_DASHSCOPE_MODEL，默认值为 qwen-max）。这些变量通过 Vite 的 loadEnv 函数从 .env 文件中加载。

### 自定义构建插件

Vite 配置中包含一个名为 copy-manifest 的自定义插件，它在构建完成后（closeBundle 钩子）执行以下操作：确保 dist 目录存在；将 manifest.json 复制到 dist 目录；将 popup/popup.css 复制到 dist/popup 目录；将 icons 目录下的所有 PNG 和 SVG 图片文件复制到 dist/icons 目录。

### 运行时依赖

项目的唯一运行时依赖是 openai 库（版本 4.28 及以上），用于与大语言模型 API 通信。这个库被用来连接阿里云百炼平台而非 OpenAI 官方接口，因为百炼平台提供了与 OpenAI 兼容的接口格式。

开发依赖包括 Chrome 扩展类型定义（@types/chrome）、Node 类型定义（@types/node）、TypeScript 编译器和 Vite 构建工具。

包管理器使用 pnpm（从 node_modules 目录结构中可以看出）。

### 构建脚本

项目定义了三个构建脚本：dev 命令启动 Vite 的监听模式构建，用于开发时实时编译；build 命令执行一次性生产构建；preview 命令用于预览构建产物。

### Chrome 扩展清单

使用 Manifest V3 规范。扩展名称为"FOMO - AI热点搜索"，描述为"一键搜索 AI/LLM/Agent 相关热门开源项目和论文"。

声明的权限包括：tabs（操作标签页）、scripting（向标签页注入脚本）、storage（Chrome 本地存储）、downloads（文件下载）、activeTab（访问当前活动标签页）。同时声明了对所有 URL 的主机权限（host_permissions 设置为 <all_urls>），这是因为智能体需要能够访问任意网站。

后台脚本配置为 Service Worker 模式，入口为 background/index.js，模块类型为 module。扩展的浏览器操作（点击图标）会弹出 popup/index.html 页面。

图标提供了三种尺寸：16 像素、48 像素和 128 像素，均为 PNG 格式，存放在 icons 目录下。

## 三、目录结构与文件命名规范

整个项目存放在 fomo 目录下，其结构如下：

根目录包含项目配置文件：package.json（包描述和依赖管理）、tsconfig.json（TypeScript 配置）、manifest.json（Chrome 扩展清单）、vite.config.ts（构建配置）。此外还有一个 types.ts 文件作为全局共享的类型定义模块，存放所有在多个模块之间共享的接口和类型。

background 目录是后台服务的代码所在。该目录的入口文件是 index.ts，其下按功能领域分为三个子目录：

- agent 子目录存放智能体相关逻辑，包含三个文件：executor.ts（执行器，即 ReAct 主循环）、prompts.ts（系统提示词和初始用户消息）、tools.ts（工具定义和工具执行器）。
- browser 子目录存放浏览器操控相关逻辑，包含两个文件：context.ts（浏览器上下文管理器，管理多标签页连接）、page.ts（单个页面的操控接口）。
- llm 子目录存放大语言模型客户端，包含一个文件：dashscope.ts（阿里云百炼 API 客户端封装）。

popup 目录是用户界面代码所在，包含三个文件：index.html（Popup 页面的 HTML 骨架）、popup.css（样式表）、popup.ts（交互逻辑）。

icons 目录存放扩展图标（PNG/SVG 文件）。

文件命名规范为全小写加连字符分隔（如 index.ts），使用 .ts 扩展名。每个子目录按照单一职责原则组织，模块文件名直接反映其功能角色。

## 四、每个模块的职责及其与其他模块的接口

### 全局类型定义模块（types.ts）

这个文件定义了在整个项目中共享的所有数据结构和类型，不包含任何业务逻辑。

它定义了以下关键数据结构：

**日志条目**（LogEntry）：记录执行过程中的每一步操作，包含时间戳（ISO 格式字符串）、类型（助手消息、工具调用、工具响应、用户消息、系统消息、错误六种之一）、步骤编号和内容（任意类型）。

**完整执行日志**（ExecutionLog）：任务的整体日志，包含任务开始时间、结束时间（可选，任务进行中时不存在）、总步骤数、结果数量和日志条目数组。

**搜索结果**（SearchResult）：单条搜索发现的资讯，包含名称/标题、信息来源 URL、产品 URL（可选）、GitHub 链接（可选）、ArXiv 链接（可选）、创新点描述、所属公司或社区、热度趋势评估、标签数组、来源平台名称和发现时间。

**智能体状态**（AgentState）：记录智能体的运行时状态，包含已收集的搜索结果数组、已访问 URL 的集合（用于去重）、当前步骤编号、最大步骤数、目标收集数量和是否完成的标志。

**执行状态**（ExecutionStatus）：一个联合字符串类型，取值为空闲、运行中、暂停、完成或错误五种之一。

**状态更新消息**（StatusUpdate）：用于在后台和 Popup 之间传递进度信息，包含执行状态、当前进度、总目标数、当前执行的动作描述（可选）和错误信息（可选）。

**消息类型**（MessageType）：定义了 Popup 和 Background 之间所有可能的通信消息格式，包括启动任务（需附带标签页 ID）、停止任务、获取状态、下载结果、下载日志、状态更新、任务完成和错误八种消息。

### 后台入口模块（background/index.ts）

这是后台 Service Worker 的入口点，负责协调所有后台模块的工作，作为 Popup 与后台逻辑之间的桥梁。

它维护了四个模块级变量：当前执行器实例（可为空）、当前搜索结果数组、当前执行状态和当前执行日志。

**对外接口**：它通过 Chrome 的消息监听器暴露了五个操作——启动任务、停止任务、获取状态、下载结果和下载日志。每个操作通过消息的 type 字段区分。

启动任务时，它首先检查是否已有任务在运行（防止重复启动），然后获取浏览器上下文单例、设置当前标签页、连接到标签页、创建智能体执行器（默认目标 20 条结果、最大 100 步）、注册状态回调、启动执行器的运行循环。运行完成后或出错时，清理浏览器上下文并置空执行器引用。

停止任务时，调用执行器的取消方法，重置状态为空闲，清理浏览器上下文。

它还监听了标签页关闭事件，当标签页被关闭时从浏览器上下文中移除对应的已连接页面。以及扩展安装/更新事件用于日志记录。

向 Popup 发送消息时（状态更新、任务完成、错误），它使用 chrome.runtime.sendMessage 方法，并在 catch 中静默忽略错误——因为 Popup 可能并未打开。

### 智能体执行器模块（background/agent/executor.ts）

这是整个项目最核心的模块，实现了 ReAct 模式的智能体循环。

**构造器**接受两个参数：目标收集数量（默认 20）和最大步骤数（默认 100）。构造时初始化智能体状态对象、消息历史（包含系统提示词和初始用户消息两条）、工具执行器实例和执行日志。

**回调机制**：提供了一个设置状态回调的方法，外部可以注册一个函数，在每次状态变更时被调用。

**主循环逻辑**（run 方法）：这是一个异步方法，返回搜索结果数组。循环的终止条件是三个之一满足：智能体状态标记为完成、外部调用了取消方法、或达到最大步骤数。

每一步的执行流程如下：

第一步，将当前的消息历史发送给大语言模型，请求模型返回下一步决策。请求中携带了所有可用工具的定义。

第二步，检查模型的响应。如果响应为空，记录错误日志并跳过本次循环。

第三步，如果模型返回了文本内容（思考过程），将前 100 个字符作为当前动作状态发送给 Popup。

第四步，将模型返回的完整消息（包含文本和工具调用信息）追加到消息历史中。

第五步，如果模型返回了工具调用请求，逐个执行。对每个工具调用：解析工具名称和参数（参数以 JSON 字符串形式给出，解析失败则使用空对象）；调用工具执行器执行操作；将工具执行结果作为 tool 角色的消息追加到消息历史中。

第六步，判断是否完成：如果模型的结束原因是"stop"且没有工具调用（意味着模型认为对话结束），则检查已收集的结果数量。如果未达到目标数量，会追加一条用户消息提示模型继续搜索或调用 finish 工具说明原因；如果已达到目标数量，标记任务完成。

第七步，消息历史管理：如果消息数组超过 50 条，执行压缩——保留第一条系统消息和最近 40 条消息，丢弃中间的旧消息。这是为了控制发送给大语言模型的上下文长度。

**错误处理**：每一步都包裹在 try-catch 中。出错时不中断循环，而是将错误信息作为用户消息追加到历史中，告诉模型"出现了错误，请尝试其他方法继续搜索"。

**取消机制**：通过设置 cancelled 标志和 isComplete 标志实现。取消后循环条件不再满足，自然退出。

**日志系统**：每个关键操作（系统消息、用户消息、助手消息、工具调用、工具响应、错误）都会被记录到执行日志中，附带时间戳和步骤编号。工具响应如果超过 5000 字符会被截断。

### 提示词模块（background/agent/prompts.ts）

负责生成发送给大语言模型的系统提示词和初始用户消息。

**日期感知**：该模块有一个内部辅助函数，获取当前日期的年、月、日以及英文月份名。这些日期信息会被动态注入到提示词中，确保智能体搜索的是最新内容。

**系统提示词**的内容包括：角色设定（专业 AI 资讯搜索智能体）、当前日期信息、目标说明（收集 10 到 20 条高质量 AI 相关资讯）、搜索策略指南（包括 Twitter/X 上的 AI 领域 KOL 账号列表、GitHub Trending、Google 搜索关键词示例、ArXiv 论文分类、Reddit 社区等五种策略）、所有可用工具的使用说明（包括每个工具的名称和用途的自然语言描述）、以及注意事项（立即保存发现、提供详细信息、避免重复、处理异常、达到目标后完成）。

搜索关键词中动态嵌入了当前年份和月份，例如"LLM agent github 2026 April"。

**初始用户消息**指示智能体从 Twitter/X 上 Andrej Karpathy 的主页开始搜索，并告知当前日期和目标数量。

模块同时导出了一个常量和一个函数：常量在模块加载时立即生成（调用函数），函数可以在需要时重新生成最新日期的提示词。

### 工具定义与执行模块（background/agent/tools.ts）

这个模块有两个职责：定义智能体可用的工具规格（供大语言模型理解），以及实现工具的实际执行逻辑。

**工具定义**部分以 OpenAI 的函数调用格式定义了 11 个工具：

- navigate：接收一个 URL 参数，导航到指定网页。
- click：接收一个 CSS 选择器参数，点击页面上的元素。
- type_text：接收选择器和文本两个参数，在输入框中输入文本。
- press_key：接收按键名称参数，模拟键盘按键。
- scroll：接收方向参数（向上或向下），滚动页面。
- get_page_content：无参数，获取当前页面的文本内容。
- get_links：无参数，获取当前页面的所有链接。
- wait：接收秒数参数（1 到 10 之间），等待指定时间。
- save_result：接收多个参数（名称、URL、产品 URL、GitHub URL、ArXiv URL、创新点、公司、热度趋势、标签数组、来源），保存一条搜索结果。其中名称、URL、创新点和来源为必填项。
- get_status：无参数，获取当前搜索进度。
- finish：接收原因参数，标记任务完成。

**工具执行器**（ToolExecutor 类）持有一个智能体状态引用，通过一个 execute 方法接收工具名称和参数，使用 switch-case 分发到对应的处理逻辑。

对于浏览器操作类工具（navigate、click、type_text、press_key、scroll、get_page_content、get_links），执行器先获取浏览器上下文单例和当前页面实例，然后委托给页面对象执行。如果页面不存在（未连接），返回错误信息字符串。

get_page_content 工具会将页面标题、URL 和内容摘要（截取前 8000 字符）组合返回。get_links 工具最多返回前 50 个链接。

wait 工具会将秒数限制在 1 到 10 之间（用 Math.min 和 Math.max 夹紧），然后执行 setTimeout 等待。

save_result 工具的逻辑最为关键：它首先从参数中构造一个搜索结果对象，对缺失的可选字段提供默认值（如公司默认为"Community"，热度默认为"Medium"，标签默认为包含"AI"的数组）。然后执行去重检查——如果已保存的结果中存在名称相同（不区分大小写）、URL 相同或 GitHub URL 相同的记录，则跳过并返回"跳过重复"的提示。通过去重检查后将结果追加到状态数组中。如果结果数量达到或超过目标数，将智能体状态标记为完成。

get_status 工具返回一个格式化的状态字符串，包含已保存数量、当前步骤和已保存项目的列表。

finish 工具设置智能体状态为完成，返回总结信息。

对于未知的工具名称，返回"未知工具"的错误信息。

### 浏览器上下文管理模块（background/browser/context.ts）

这个模块负责管理多个浏览器标签页的连接生命周期，是浏览器操控的中间协调层。

**BrowserContext 类**维护两个私有状态：当前活动标签页的 ID 和一个从标签页 ID 到 Page 实例的映射表。

它提供以下能力：

获取或创建页面：根据标签页 ID 查找已存在的 Page 实例，不存在则创建新实例。

连接页面：将一个 Page 实例加入映射表，调用 Page 的 attach 方法。如果已存在则直接返回成功。

获取当前页面：根据当前标签页 ID 获取对应的 Page 实例。如果不存在，自动创建并连接。

切换标签页：通过 Chrome 的 tabs API 激活指定标签页，获取或创建对应的 Page 实例，更新当前标签页 ID。

导航到 URL：如果当前页面已连接，通过 Page 实例导航。如果未连接，使用 Chrome 的 tabs API 直接更新标签页 URL，等待加载完成后重新创建和连接 Page 实例。

等待标签页加载：通过监听 Chrome 的 tabs.onUpdated 事件，等待指定标签页的 status 变为 complete。设置了 30 秒的超时保护——如果 30 秒内未收到加载完成事件，自动解除等待。

清理：遍历所有已连接的页面实例，逐一断开连接，清空映射表和当前标签页 ID。

移除已连接页面：断开并移除指定标签页的 Page 实例，如果被移除的是当前标签页则置空 ID。

**单例模式**：模块底部维护一个模块级变量，通过一个工厂函数确保全局只有一个 BrowserContext 实例。这个工厂函数是其他模块获取浏览器上下文的唯一入口。

### 页面操控模块（background/browser/page.ts）

这个模块封装了对单个浏览器标签页的所有操控能力，使用 Chrome Extension API（而非传统的 DevTools Protocol）。

**Page 类**的构造器接收一个标签页 ID。

由于使用的是 Chrome 扩展的 scripting API 而非 CDP 协议，attach 和 detach 方法实际上是空操作（no-op）——attached 属性始终返回 true。这种设计保留了接口统一性，如果未来需要切换到 CDP 协议可以方便修改。

**页面导航**：通过 chrome.tabs.update 方法更新标签页 URL，然后调用内部的等待加载方法。等待加载方法使用与 BrowserContext 相同的机制——监听 tabs.onUpdated 事件等待 status 变为 complete，额外等待 500 毫秒让页面渲染稳定，超时时间为 30 秒。

**获取页面信息**：通过 chrome.tabs.get 获取标签页元数据（URL 和标题）。

**脚本注入执行**：所有需要与页面内容交互的操作都通过 chrome.scripting.executeScript 实现，这是一个通用的私有方法，接收一个函数并在目标标签页的上下文中执行。

**点击操作**：在目标页面中用 CSS 选择器查找元素，找到后调用 click 方法。找不到元素时返回包含选择器的错误信息。点击后等待 500 毫秒让页面响应。

**文本输入**：查找输入框元素，设置焦点，直接赋值 value 属性，然后派发 input 和 change 两个冒泡事件以触发前端框架的响应式更新。

**按键模拟**：创建 KeyboardEvent 派发到当前焦点元素。对于 Enter 键有特殊处理——会查找焦点元素所在的表单，如果存在则派发 submit 事件。按键后等待 500 毫秒。

**页面滚动**：向上或向下滚动 500 像素，滚动后等待 300 毫秒。

**获取页面文本**：克隆 body 元素（避免修改原页面），移除 script、style 和 noscript 标签，使用 TreeWalker 遍历所有文本节点，收集非空文本并用空格连接，最终截取前 15000 字符返回。

**获取页面链接**：收集所有带 href 属性的 a 标签，提取链接文本（截取前 100 字符）和 href 属性值，过滤掉缺失 href 或文本的链接。

**等待选择器**：一个轮询机制，每 500 毫秒检查一次目标选择器对应的元素是否出现在 DOM 中。默认超时 10 秒。

所有操作方法都返回描述性的字符串结果（成功或失败的消息），方便大语言模型理解操作结果。所有操作都包含 try-catch 错误处理，错误信息会被格式化为可读字符串返回，而非抛出异常。

### 大语言模型客户端模块（background/llm/dashscope.ts）

封装了与阿里云百炼平台通信的逻辑。

**DashScopeClient 类**内部使用 OpenAI 官方的 JavaScript SDK 创建客户端实例，但将 API Key 和 Base URL 指向阿里云百炼平台。构造时会检查 API Key 是否存在，如果为空则抛出错误，错误信息提示用户在 .env.local 文件中设置 VITE_DASHSCOPE_API_KEY。

客户端创建时设置了 dangerouslyAllowBrowser 为 true，因为 Chrome 扩展的后台脚本虽然运行在浏览器环境中，但 Service Worker 的安全模型与普通网页不同，OpenAI SDK 默认会阻止在浏览器中使用（防止泄露 API Key），需要显式启用。

**chat 方法**：接收消息数组和可选的工具定义数组，调用 OpenAI 兼容的聊天补全接口。如果提供了工具定义，tool_choice 设置为 auto（让模型自行决定是否调用工具）。返回完整的模型响应对象。

**complete 方法**：简化接口，接收单个字符串提示词，返回模型回复的文本内容。

**单例模式**：与 BrowserContext 一样，通过模块级变量和工厂函数实现懒加载的单例。

### Popup HTML 页面（popup/index.html）

定义了弹出窗口的 DOM 结构，使用中文 HTML 标签（lang 属性为 zh-CN）。

页面结构从上到下依次为：

顶部栏包含"FOMO"品牌文字和一个状态指示点（通过 CSS 类名切换不同颜色来表示空闲、运行中、完成、错误四种状态）。

中间是一个 SVG 实现的环形进度条，圆环半径为 52，周长约 327。进度通过 stroke-dashoffset 属性控制填充比例。圆环中心显示当前进度数字和"/20"的总量文字。

进度环下方是当前动作描述文字区域，初始显示"Ready"。

底部是四个圆形控制按钮，每个按钮内嵌一个 SVG 图标：开始按钮（播放三角形图标）、停止按钮（方块图标，默认禁用）、下载结果按钮（下载箭头图标，默认禁用）、下载日志按钮（文档图标，默认禁用）。

最底部是一个默认隐藏的错误提示区域。

### Popup 样式模块（popup/popup.css）

采用 CSS 变量（自定义属性）定义主题色系：主背景白色、次要背景浅灰色、强调色绿色（#10b981）、主要文字近黑色、次要文字灰色、危险色红色、警告色琥珀色。

弹窗固定宽度 240 像素，最小高度 280 像素，使用系统字体栈。

状态指示点大小 8 像素，运行中状态有绿色光晕和脉冲动画（1.5 秒周期的透明度和缩放变化）。

环形进度条的填充动画使用 0.5 秒缓动过渡。进度数字使用 32 像素轻量字体。

控制按钮为 36 像素直径的圆形，悬停时放大 5%，按下时缩小 5%。禁用状态时透明度降低到 30%。不同按钮有不同的颜色主题：开始按钮为绿色底白色图标、停止按钮为红色底白色图标、下载结果按钮为灰底绿色边框、下载日志按钮为灰底琥珀色边框。

当前动作文字使用 11 像素字号，超长文本会被省略号截断。

错误消息使用红色文字和淡红色背景，圆角 6 像素。

### Popup 交互逻辑模块（popup/popup.ts）

这是 Popup 页面的唯一脚本文件，负责所有的 UI 交互和与后台的通信。

**初始化**：在模块顶部，通过 getElementById 获取所有 DOM 元素的引用——状态指示点、进度环 SVG 圆形元素、进度数字、当前动作文字、四个按钮和错误消息区域。定义了环形周长常量 327（等于 2π × 52）和总目标数 20。维护一个搜索结果缓存和日志可用性标志。

**进度环更新**：通过计算 strokeDashoffset = 周长 - (进度 / 总数) × 周长 来控制进度环的填充比例，同时更新数字显示。

**UI 状态更新**：接收一个状态更新对象，根据执行状态切换按钮的启用/禁用状态和样式：

- 运行中：开始按钮禁用，停止按钮启用，两个下载按钮禁用。
- 完成：开始按钮启用，停止按钮禁用，两个下载按钮启用。
- 错误：开始按钮启用，停止按钮禁用，下载结果按钮根据缓存是否有数据决定，显示错误消息。
- 空闲或其他：开始按钮启用，停止按钮禁用，下载按钮根据缓存和日志状态决定。

**启动任务**：先查询当前活动标签页，检查是否为 chrome:// 或 chrome-extension:// 开头的内部页面（这些页面无法注入脚本）。如果是则抛出错误。否则向后台发送 START_TASK 消息并附带标签页 ID，同时立即更新 UI 为运行状态。

**停止任务**：向后台发送 STOP_TASK 消息，更新 UI 为空闲状态，显示"Stopped"。

**下载结果**：将缓存的搜索结果序列化为美化的 JSON 字符串，创建 Blob 对象，通过 Chrome 的 downloads API 触发下载，文件名格式为 fomo-results-年-月-日T时-分-秒.json，弹出保存对话框让用户选择存储位置。

**下载日志**：向后台发送 DOWNLOAD_LOGS 消息请求日志数据，收到响应后以类似方式下载 JSON 文件。

**消息监听**：监听来自后台的三种消息——STATUS_UPDATE（更新 UI）、TASK_COMPLETE（缓存结果并更新 UI 为完成状态）、ERROR（更新 UI 为错误状态并显示错误信息）。

**页面加载时**：立即向后台发送 GET_STATUS 消息获取当前状态（用于 Popup 被关闭后重新打开的场景），并为四个按钮绑定点击事件。

## 五、模块间的数据流

整个系统的数据流可以从用户操作开始追溯：

**任务启动流程**：用户点击 Popup 的开始按钮 → Popup 查询活动标签页获取 tabId → Popup 通过 chrome.runtime.sendMessage 发送 START_TASK 消息给 Background → Background 入口模块接收消息 → 获取 BrowserContext 单例并设置 tabId → 调用 BrowserContext 的 switchTab 激活标签页并创建 Page 实例 → 创建 AgentExecutor 实例（传入目标数和最大步数） → 注册状态回调（将状态更新转发给 Popup） → 调用 AgentExecutor 的 run 方法启动 ReAct 循环。

**ReAct 循环数据流**：AgentExecutor 构造初始消息数组（系统提示词 + 初始用户消息） → 调用 DashScopeClient 的 chat 方法，将消息数组和工具定义发送给阿里云百炼 API → 百炼 API 返回模型响应（包含文本推理和/或工具调用请求） → AgentExecutor 解析响应 → 如果有工具调用，逐个传给 ToolExecutor → ToolExecutor 根据工具名称分发到对应处理逻辑 → 浏览器操控类工具通过 BrowserContext 获取 Page 实例 → Page 实例通过 chrome.scripting.executeScript 在目标标签页中执行 JavaScript → 执行结果沿原路返回到 ToolExecutor → 工具结果作为 tool 角色消息追加到消息数组 → 进入下一次循环。

**结果保存数据流**：当大语言模型决定保存一条结果时，调用 save_result 工具 → ToolExecutor 从参数中构造 SearchResult 对象 → 执行去重检查 → 通过检查则追加到 AgentState 的 results 数组中 → 如果达到目标数量则设置 isComplete 标志 → AgentExecutor 通过状态回调通知 Background → Background 通过 chrome.runtime.sendMessage 通知 Popup → Popup 更新进度环。

**任务完成数据流**：ReAct 循环退出 → AgentExecutor 返回结果数组 → Background 入口模块缓存结果 → 通过消息通知 Popup → Popup 缓存结果并更新 UI 为完成状态 → 用户点击下载按钮 → Popup 将缓存数据序列化为 JSON 并触发文件下载。

**状态查询数据流**：Popup 打开时发送 GET_STATUS → Background 返回当前状态对象（包含执行状态和进度） → Popup 据此更新 UI。这确保了 Popup 在被关闭后重新打开时能恢复到正确的状态显示。

## 六、关键业务逻辑的自然语言描述

### ReAct 智能体循环

整个系统的核心创新在于用大语言模型驱动一个完全自主的网页浏览器操控循环。其算法本质是：

给模型一个明确的任务目标（搜集 AI 热点资讯），赋予它一组浏览器操控工具，然后让模型自行决定访问哪些网站、如何在页面上寻找信息、何时保存结果、何时切换策略。模型的每次"思考"输出都被追加到对话历史中，形成链式推理。模型的每次"行动"（工具调用）的结果也被追加到历史中，为后续推理提供上下文。

这种设计使得智能体能够应对动态的网页环境：如果某个页面无法访问，模型可以自行决定尝试其他网站；如果页面结构不符合预期，模型可以调整选择器或改用 get_page_content 提取文本。

### 搜索策略与优先级

系统提示词中预设了五种搜索策略，按推荐优先级排列：首先是 Twitter/X 上的 AI 领域意见领袖（Karpathy、Andrew Ng、Yann LeCun 等），其次是 GitHub Trending，再次是 Google 搜索，然后是 ArXiv 论文列表，最后是 Reddit 社区。初始用户消息强制要求从 Karpathy 的 Twitter 主页开始，但之后模型可以自由选择和组合策略。

### 去重机制

保存结果时执行三维度去重：项目名称（忽略大小写）、信息来源 URL、GitHub URL。任一维度匹配已有记录即判定为重复并跳过。虽然 AgentState 中定义了 visitedUrls 集合用于追踪已访问的 URL，但在当前实现中这个集合并未被使用，去重完全依赖 save_result 工具中的结果级别去重。

### 消息历史压缩

为了控制发送给大语言模型的上下文长度（避免超过模型的上下文窗口和产生不必要的费用），当消息数组超过 50 条时会执行压缩：保留第一条系统提示词消息不动，丢弃早期的对话历史，只保留最近 40 条消息。这意味着大约每 10 步执行一次压缩。

### 自动追问机制

当大语言模型在未达到目标数量时自行停止对话（finish_reason 为 stop 且没有工具调用），系统不会直接接受，而是注入一条用户消息提醒模型还需要继续搜索，告知当前进度和目标差距。只有当模型显式调用 finish 工具或结果数量达到目标时，任务才会真正结束。

### 进度环算法

环形进度条使用 SVG 的 stroke-dasharray 和 stroke-dashoffset 技术。圆环半径 52，周长约 327（2π × 52）。dasharray 固定为 327（即整个圆周），dashoffset 从 327（完全空）递减到 0（完全填满）。偏移量 = 327 - (当前进度 / 20) × 327。

## 八、错误处理策略

### 大语言模型调用层面

ReAct 循环的每一步都被 try-catch 包裹。如果大语言模型的 API 调用失败（网络错误、速率限制、模型不可用等），错误会被捕获并转化为一条用户角色的消息追加到对话历史中，内容为"出现错误：[错误信息]。请尝试其他方法继续搜索。"这样模型在下一轮循环中会感知到错误并调整策略，而非整个任务崩溃。

如果模型返回空响应（choices 数组为空或第一个 choice 不存在），记录错误日志后跳过本步继续循环。

工具参数的 JSON 解析如果失败，使用空对象作为参数继续执行，不中断流程。

### 浏览器操控层面

Page 类的每个操控方法都有独立的 try-catch。所有错误都被转化为描述性的字符串返回（如"点击失败：[错误信息]"、"导航失败：[错误信息]"），而非抛出异常。这是因为这些返回值会作为工具结果发送给大语言模型，模型需要能理解操作失败并做出相应调整。

页面加载等待设置了 30 秒超时保护。如果超时，不抛出异常而是继续执行——可能页面已经部分加载，足够提取信息。

脚本注入执行如果失败（可能因为页面安全策略限制），返回 null 或空数组。

### 后台入口层面

startTask 函数有全局 try-catch。如果标签页连接失败或执行器运行失败，错误状态和错误消息会通过消息通道发送给 Popup 显示。无论成功还是失败，finally 块确保浏览器上下文被清理、执行器引用被置空。

向 Popup 发送消息的操作会 catch 错误并静默忽略，因为 Popup 可能未打开。

### Popup 层面

启动任务时检查当前标签页是否为 Chrome 内部页面（chrome:// 或 chrome-extension:// 开头），如果是则显示"Cannot run on this page"错误消息。

查询活动标签页失败或标签页 ID 不存在时抛出"No active tab"错误，被 catch 后显示在错误消息区域。

### DashScope 客户端层面

构造时如果 API Key 为空，立即抛出带有配置指引的错误。这是唯一一个会导致任务无法启动的致命错误。

## 九、配置和默认值
